# Data Model & Migrations (Phase 2) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build the complete Supabase Postgres schema for the IHS Scheduling Tool — 23 tables covering every Section 2 domain entity, helper SQL functions, and the full Section 4.3 RLS capability matrix — and prove it on a hosted Supabase project.

**Architecture:** Five thematic SQL migrations (`core`, `schedule`, `collaboration`, `rls_functions`, `rls_policies`) applied to a hosted Supabase project via the Supabase CLI. RLS is enabled on every table; access is decided by `SECURITY DEFINER` helper functions reading a data-driven `role_capabilities` table. A `BEFORE UPDATE` trigger enforces the one column-level rule RLS cannot express (external users may edit progress fields only). A `seed.sql` and a `psql`-run verification script prove the internal/external split.

**Tech Stack:** Supabase (Postgres 15), Supabase CLI v2.101.0, SQL. No application code, no `@supabase/*` packages — that is Phase 3.

**Source spec:** `docs/superpowers/specs/2026-05-22-data-model-design.md`

---

## Conventions & Working Notes

- **Authoritative reference:** invoke the `supabase` skill for any uncertainty about CLI commands or Supabase-specific SQL.
- **Migration files** are created with `supabase migration new <name>`, which prepends a timestamp. This plan refers to them by logical name (`core`, `schedule`, …); the real filenames will be `<timestamp>_core.sql` etc. Edit the generated file — never hand-name migrations.
- **Forward-only correction:** `supabase db push` records applied migrations and will not re-apply an edited file. If you must change an already-pushed migration during development, run `supabase db reset --linked` to wipe and re-apply the full corrected set. The hosted project is greenfield with no real data, so this is safe.
- **Two spec refinements made in this plan** (strict improvements; the spec's field lists are the design intent):
  1. The spec's `constraints` table is named **`activity_constraints`** — `constraints` is a SQL reserved word.
  2. The spec's separate `can()` / `cap_scope()` helpers are implemented as a single **`has_capability(capability, project, is_responsible)`** function, because a capability granted with `scope = 'responsible'` (e.g. a Trade Partner Editor's `update_progress`) must NOT pass a plain existence check — it must also test responsibility. A single function prevents that class of bug.
  3. The spec §6 lists two seed users; this plan seeds **three** (adds a Trade Partner Viewer) so verification §7 check 4 (TP Viewer is read-only) is actually exercisable.

-- Helper functions. SECURITY DEFINER so policies stay thin and reads of
-- memberships/role_capabilities inside them are not re-filtered by RLS
-- (which would recurse). All set an explicit search_path.

create or replace function current_company_type()
returns company_type
language sql stable security definer set search_path = public
as $$
  select c.type
  from users u
  join companies c on c.id = u.company_id
  where u.id = auth.uid();
$$;

create or replace function is_member(p_project uuid)
returns boolean
language sql stable security definer set search_path = public
as $$
  select exists (
    select 1 from memberships m
    where m.user_id = auth.uid() and m.project_id = p_project
  );
$$;

create or replace function role_on(p_project uuid)
returns project_role
language sql stable security definer set search_path = public
as $$
  select m.role from memberships m
  where m.user_id = auth.uid() and m.project_id = p_project;
$$;

-- Returns true only if the caller's project role grants the capability AND
-- the grant's scope is satisfied: scope 'all' always; scope 'responsible'
-- only when p_is_responsible is true. No matching row -> false (least
-- privilege). This single function replaces the spec's can()/cap_scope().
create or replace function has_capability(
  p_capability text, p_project uuid, p_is_responsible boolean default false)
returns boolean
language sql stable security definer set search_path = public
as $$
  select exists (
    select 1 from role_capabilities rc
    where rc.role = role_on(p_project)
      and rc.capability = p_capability
      and (rc.scope = 'all' or (rc.scope = 'responsible' and p_is_responsible))
  );
$$;

create or replace function is_responsible(p_activity uuid)
returns boolean
language sql stable security definer set search_path = public
as $$
  select exists (
    select 1
    from activities a
    join users u on u.id = auth.uid()
    where a.id = p_activity
      and a.responsible_company_id = u.company_id
  );
$$;

-- Column-level guard: an external user may change progress fields only.
-- Compares old/new as jsonb so any future column is locked by default.
create or replace function enforce_external_progress_only()
returns trigger
language plpgsql security definer set search_path = public
as $$
declare
  allowed text[] := array[
    'percent_complete','actual_start','actual_finish',
    'remaining_duration','version','updated_at'];
  old_j jsonb := to_jsonb(old);
  new_j jsonb := to_jsonb(new);
  k text;
begin
  if current_company_type() = 'external' then
    for k in select jsonb_object_keys(old_j) loop
      if not (k = any(allowed))
         and (old_j -> k) is distinct from (new_j -> k) then
        raise exception

      end if;
    end loop;
  end if;
  return new;
end;
